# Test Fixtures — What They Are and How to Read the Output Files

---

## Background: what sigQC is actually doing

To understand the fixtures you first need to understand the two things sigQC works with.

**A gene expression matrix** is a table of measurements from a sequencing experiment. Rows are genes (e.g. `gene_1`, `TP53`, `BRCA1`), columns are samples (e.g. patients, cells), and each cell is a number representing how much that gene is expressed in that sample. A real dataset might have 20,000 genes and 500 patients.

**A gene signature** is simply a list of genes chosen by a researcher because they believe those genes together represent some biological concept — for example, "these 8 genes are all upregulated during cell proliferation". The researcher wants to know: does this signature actually work as a coherent unit when applied to a real dataset? Are the genes in the list co-expressed? Are they consistently expressed across samples? sigQC computes a set of quality metrics to answer these questions.

So the inputs to sigQC are:
1. One or more expression matrices (the data)
2. One or more gene signatures (the lists of genes to evaluate)

And the output is a summary of how "good" each signature looks in each dataset.

---

## What a fixture is

The tests cannot use real patient data (it is large, private, and would make tests slow and non-reproducible). Instead, the tests use **fixtures**: small synthetic datasets generated by an R script with a fixed random seed. The same script always produces the same numbers. Anyone can regenerate them, and every test run uses the same data.

The script that generates the small fixture is `sigQC-master/tests/fixtures/fixture_generator.R`. You run it with:

```bash
cd sigQC-master
Rscript tests/fixtures/fixture_generator.R
```

It uses `set.seed(42)` at the top, so the output is deterministic.

## Small fixture — what was generated and why

### The expression matrices (the "datasets")

The script creates two fake expression matrices, called `dataset_A` and `dataset_B`. Each has **20 genes × 10 samples** — tiny compared to real data, but enough to test the logic.

**How `dataset_A` was made:**

```r
base_signal <- rnorm(10, mean=5, sd=2)   # one random vector of 10 numbers
# genes 1-4: each is base_signal + a little noise
dataset_A[1,] <- base_signal + rnorm(10, sd=0.5)
dataset_A[2,] <- base_signal + rnorm(10, sd=0.5)
...
# genes 5-20: each is completely independent random noise
dataset_A[5,] <- rnorm(10, mean=runif(1,2,8), sd=runif(1,0.5,3))
...
# One cell is set to NA manually
dataset_A[3, 5] <- NA
```

This means `gene_1` through `gene_4` are all driven by the same `base_signal` — they will be highly correlated with each other. `gene_5` through `gene_20` are independent — they will not correlate with anything. The single NA at `gene_3, sample_5` is injected deliberately to test that the code handles missing values correctly and does not crash or silently ignore them.

**How `dataset_B` was made:**

Same structure, but with a different `base_signal_B` (different batch/experiment). One additional change:

```r
dataset_B[5,] <- rep(4.0, 10)   # gene_5 is constant: 4.0 in every sample
```

`gene_5` has zero variance in `dataset_B` — the same value in every single sample. This deliberately triggers the BUG-2 scenario: when you try to z-transform a gene with zero standard deviation you divide by zero, producing `NaN` or `Inf`. The test verifies the fix is in place. See `sigQC-master/AUDIT.md § BUG-2` for the full explanation.

### The gene signatures (the "signatures")

A signature is just a list of gene names. The fixture defines two:

**`compact_sig`**: `gene_1, gene_2, gene_3, gene_4, gene_missing`

These are the four correlated genes from the dataset construction above, plus one gene called `gene_missing` that does not appear in either dataset at all. The signature is called "compact" because its genes are designed to move together — they should score high on co-expression metrics.

The `gene_missing` entry is there to test a specific scenario: a researcher might include a gene in their signature list that was not measured in a particular experiment. The code should handle this gracefully (skip it, report it as missing) rather than crashing.

**`diffuse_sig`**: `gene_5, gene_10, gene_12, gene_15, gene_18`

These are all from the independent/random genes. They have no shared signal. The signature is called "diffuse" because its genes do not co-express — they should score low on co-expression metrics.

The two signatures together form a **positive control** (`compact_sig`) and a **negative control** (`diffuse_sig`). Any metric that claims to measure co-expression should produce a higher value for `compact_sig` than for `diffuse_sig`. If the ordering is reversed, the metric is broken.

### Reading `fixture_signatures.csv`

```
signature,   gene
compact_sig, gene_1
compact_sig, gene_2
compact_sig, gene_3
compact_sig, gene_4
compact_sig, gene_missing
diffuse_sig, gene_5
diffuse_sig, gene_10
diffuse_sig, gene_12
diffuse_sig, gene_15
diffuse_sig, gene_18
```

This is just the two signature lists in "long format" — one row per gene, with a column saying which signature it belongs to. Python reads this as a DataFrame and groups by `signature` to reconstruct the lists.

### Reading `fixture_dataset_A.csv` (and `dataset_B`)

This is the expression matrix as a CSV. The first column is the row index (gene names), the remaining columns are samples:

```
         sample_1  sample_2  sample_3  ...  sample_10
gene_1     6.12      5.89      4.78    ...    5.23
gene_2     6.08      5.91      4.81    ...    5.19
gene_3     6.15      5.84      NA      ...    5.27
...
gene_5     3.21      7.44      1.89    ...    5.52
```

Notice `gene_1` and `gene_2` have similar values across samples (both track the same base signal). `gene_5` is random. `gene_3` has a blank cell (NA) in `sample_5`.

### `fixture_small.rds`

This is the same data but bundled into a single R binary object. R tests load it with `readRDS()` so they don't need to parse CSVs. Python tests use the CSVs instead.

## Medium fixture — what was generated and why

The small fixture tests correctness on edge cases. The medium fixture tests **statistical correctness at a more realistic scale**: 100 genes × 50 samples × 3 datasets, with 5 signatures.

The five signatures are designed to cover a range of signature quality:

| Signature | What it models |
|-----------|---------------|
| `tight_coexpr` | 10 genes all generated from the same latent signal with tiny noise — maximum possible co-expression |
| `loose_coexpr` | 10 genes with moderate correlation |
| `mixed_quality` | Mix of correlated and random genes — realistic case where a signature has some good and some poor genes |
| `high_var` | Genes with high variance and skewed distributions — tests variability metrics |
| `random_baseline` | 10 completely random genes — null/negative control, should score low on everything |

The three datasets represent different experimental batches. The same signature should ideally behave consistently across all three — disagreement across datasets is itself a quality signal.

Files produced: `fixture_medium.rds`, `fixture_medium_dataset_A.csv`, `fixture_medium_dataset_B.csv`, `fixture_medium_dataset_C.csv`, `fixture_medium_signatures.csv`. Same format as the small fixture files.

## Reference output files — the R ground truth

After generating the fixtures, you run a second R script (`generate_reference_outputs.R`) that runs the **refactored R code** on those fixtures and saves the results as CSVs. These CSVs are the ground truth that the Python tests compare against.

```bash
Rscript tests/fixtures/generate_reference_outputs.R
# creates: tests/fixtures/reference_outputs/*.csv
```

### Naming convention

```
{module}_radar_{signature}_{dataset}.csv
```

For example: `compact_radar_compact_sig_dataset_A.csv` contains the compactness metrics that the R code produced for `compact_sig` in `dataset_A`.

Each file contains **one row** with one column per metric that module produces. When a Python test calls `test_cross_validation`, it loads this file and checks that the Python output matches the R value within `rtol=1e-4`.

### What each module's metrics mean

**`var_radar_*` — variability metrics**

How variable are the signature genes compared to the rest of the genome?

| Column | What it means |
|--------|---------------|
| `sd_median_ratio` | Median standard deviation of signature genes ÷ median SD of all genes. Value of 1.0 means signature genes are exactly as variable as average. Higher = more variable than average. |
| `coeff_of_var_ratio` | Same but using CV (SD/mean) instead of SD — this normalises for expression level, so a highly-expressed gene with large SD is treated the same as a lowly-expressed gene with proportionally large SD. |
| `abs_skewness_ratio` | Median absolute skewness of signature genes ÷ all genes. High skewness suggests non-normal (potentially bimodal) expression, which can indicate a gene switches between "on" and "off" states. |
| `prop_top_10_percent` | Fraction of signature genes in the top 10% most variable genes. A value of 0.8 means 80% of your signature genes are in the top 10% of the genome by variability. |
| `prop_top_25_percent` | Same for top 25%. |
| `prop_top_50_percent` | Same for top 50%. |

All values are in [0, 1] — they represent proportions or normalised ratios clipped to that range for the radar chart.

**`expr_radar_*` — expression metrics**

Are the signature genes actually expressed (detected) in the samples?

| Column | What it means |
|--------|---------------|
| `med_prop_na` | Median fraction of samples where the gene's expression is missing (NA). A gene absent from the assay entirely would have `med_prop_na = 1.0`. Low is good. |
| `med_prop_above_med` | Median fraction of samples where the gene's expression exceeds the dataset-wide median expression level. This measures "is this gene actually expressed above background?" High is good. |

**`compact_radar_*` — compactness metric**

Do the signature genes move together across samples?

| Column | What it means |
|--------|---------------|
| `autocor_median` | Median pairwise Spearman correlation between all pairs of signature genes. For `compact_sig` with 4 correlated genes this should be near 1. For `diffuse_sig` with random genes this should be near 0. This is the most direct measure of whether the signature is a coherent biological unit. |

**`stan_radar_*` — standardisation metric**

Does z-transforming the genes change which samples score high vs low?

| Column | What it means |
|--------|---------------|
| `standardization_comp` | Spearman correlation between sample scores computed from raw expression and from z-transformed expression. Near 1 = the signature's sample ranking is robust to expression scale. Near 0 = the ranking depends heavily on whether you normalise, which means the signature may be driven by absolute expression levels rather than relative patterns. |

**`metrics_radar_*` — PCA and scoring agreement metrics**

Do the three ways of scoring samples (mean, median, PCA first component) agree with each other?

| Column | What it means |
|--------|---------------|
| `rho_mean_med` | Spearman correlation between mean-score and median-score across all samples. Near 1 = the two aggregation methods agree; outliers are not distorting the mean. |
| `rho_pca1_med` | Spearman correlation between PCA1 scores and median scores. Near 1 = the first principal component of the signature gene matrix captures the same biological variation as the median score. |
| `rho_mean_pca1` | Spearman correlation between mean scores and PCA1 scores. |
| `prop_pca1_var` | Fraction of variance explained by the first principal component of the signature gene expression sub-matrix. Near 1 = the signature is dominated by a single coherent signal. Near 0 = the genes vary independently, suggesting they do not form a coherent signature. |

### The full radar table: `radar_output_table.csv`

This is the final summary of the entire pipeline run — one row per (signature, dataset) combination, 14 columns (all metrics above combined). The row index is `{signature}_{dataset}`, e.g. `compact_sig_dataset_A`.

```
                          sd_median_ratio  ...  autocor_median  ...  prop_pca1_var
compact_sig_dataset_A          0.82        ...      0.94        ...     0.91
compact_sig_dataset_B          0.79        ...      0.96        ...     0.93
diffuse_sig_dataset_A          0.41        ...      0.12        ...     0.23
diffuse_sig_dataset_B          0.38        ...      0.09        ...     0.21
```

A row with values close to 1 across all columns = high-quality, coherent signature in that dataset. The `compact_sig` rows should be consistently higher than `diffuse_sig` rows on compactness metrics. The radar chart is a visual representation of this table — each metric becomes one spoke of the radar, and the enclosed area represents overall signature quality.

**Known limitation:** the area formula in the R code is missing a `sin()` factor — it is a proportional proxy, not a true geometric area. Ordinal comparisons (larger = better) are still valid. See `sigQC-master/AUDIT.md § ISSUE-3`.
