# Test Fixtures — Design and Output Format

This document explains what the synthetic test fixtures are modelling, why they are designed the way they are, and how to read every output file they produce.

---

## What a "fixture" is

A fixture is a small, fully synthetic, deterministic dataset — no real gene expression data, no patient data. It is generated once by an R script and committed to the repository so that every test run on every machine uses exactly the same numbers.

The fixtures exist for two reasons:
1. **Correctness**: run the Python code and the R code on the same data; the outputs should be numerically identical (within floating-point tolerance).
2. **Edge cases**: deliberately inject situations — NA values, constant genes, missing genes — that expose known bugs and verify they are fixed.

---

## Small fixture (`fixture_generator.R`)

**Size:** 20 genes × 10 samples, 2 datasets, 2 signatures.

### Datasets

| Dataset | What makes it special |
|---------|----------------------|
| `dataset_A` | One NA value at `gene_3, sample_5`. Tests that NA propagation is handled correctly (not silently dropped, not crashing). |
| `dataset_B` | `gene_5` has constant expression (`rep(4.0, 10)`). Tests the BUG-2 fix: z-transform of a zero-variance gene must produce 0, not `NaN`/`Inf`. See `sigQC-master/AUDIT.md § BUG-2`. |

### Signatures

| Signature | Genes | What it models |
|-----------|-------|---------------|
| `compact_sig` | `gene_1`–`gene_4` + `gene_missing` | Four genes that are **correlated** (all generated from the same base signal + small noise). `gene_missing` does not exist in either dataset. Tests that missing genes are handled gracefully (excluded from analysis, not causing a crash). |
| `diffuse_sig` | `gene_5`, `gene_10`, `gene_12`, `gene_15`, `gene_18` | Five genes with **no correlation** (each independently random). `gene_5` is the constant gene in `dataset_B`. |

The key property: `compact_sig` has high co-expression, `diffuse_sig` does not. Tests that compare these two should always find `compact_sig > diffuse_sig` on compactness metrics. If that ordering reverses, the metric is broken.

### Files produced

| File | Contents |
|------|----------|
| `fixture_small.rds` | All of the above as a single R list object. Loaded by R tests via `readRDS()`. |
| `fixture_dataset_A.csv` | Expression matrix, rows = genes (`gene_1`…`gene_20`), columns = samples (`sample_1`…`sample_10`). One cell is blank (NA). |
| `fixture_dataset_B.csv` | Same shape. Row `gene_5` is all `4.0`. |
| `fixture_signatures.csv` | Long-format table: two columns `signature` and `gene`. Five rows per signature. Loaded by Python tests via `conftest.py`. |

## Medium fixture (`fixture_medium_generator.R`)

**Size:** 100 genes × 50 samples, 3 datasets, 5 signatures.

This fixture is **not** about edge cases — it is about statistical correctness at a more realistic scale. The signals are strong enough that comparisons against R reference values are meaningful.

### Datasets

| Dataset | What makes it special |
|---------|----------------------|
| `dataset_A` | Clean data with batch offset (`mean ~ 5`). |
| `dataset_B` | Different batch offset (`mean ~ 6`). One gene has scattered NAs. |
| `dataset_C` | Third batch. One gene is constant (zero-variance). Some genes missing entirely. |

Three datasets are used because several metrics are more informative when evaluated across batches — e.g., does a signature remain compact across different experimental conditions?

### Signatures

| Signature | What it models | Why it exists |
|-----------|---------------|---------------|
| `tight_coexpr` | 10 genes all generated from the same latent signal with tiny noise | Highest possible compactness; serves as a positive control for `autocor_median` |
| `loose_coexpr` | 10 genes with moderate correlation | Intermediate compactness |
| `mixed_quality` | Mix of correlated and random genes | Tests that the metric is not dominated by a few well-behaved genes |
| `high_var` | Genes with high variance and skewed distributions | Tests `sd_median_ratio`, `abs_skewness_ratio` |
| `random_baseline` | 10 purely random genes | Negative control — should score low on every quality metric |

The ordering `tight_coexpr > loose_coexpr > random_baseline` on compactness metrics, and `high_var > random_baseline` on variability metrics, are implicit invariants.

### Files produced

| File | Contents |
|------|----------|
| `fixture_medium.rds` | Full R list, loaded by R tests. |
| `fixture_medium_dataset_{A,B,C}.csv` | Expression matrices (100 genes × 50 samples). |
| `fixture_medium_signatures.csv` | Long-format table: columns `signature` and `gene`. |

## Reference output files

Reference outputs are the **R ground-truth** values for each metric. They are generated by running the refactored R code (`R_refactored/`) on the fixtures and saving the results as CSVs. Python tests load these CSVs and compare their outputs against them.

### Naming convention

```
{module_prefix}_radar_{signature_name}_{dataset_name}.csv
```

Each file contains a **single row** with one column per radar metric that the module produces.

### Module prefixes and what they measure

| Prefix | Module | Metrics in the file | What the metric means |
|--------|--------|--------------------|-----------------------|
| `var_radar_*` | `eval_var` | `sd_median_ratio` | Ratio of the median SD of signature genes to the median SD of all genes. High value = signature genes are more variable than average. |
| | | `coeff_of_var_ratio` | Same idea but using coefficient of variation (SD/mean), which normalises for expression level. |
| | | `abs_skewness_ratio` | Ratio of median absolute skewness of signature genes to all genes. High skewness = non-normal, potentially bimodal expression. |
| | | `prop_top_10_percent` | Proportion of signature genes in the top 10% most variable genes in the dataset. |
| | | `prop_top_25_percent` | Same for top 25%. |
| | | `prop_top_50_percent` | Same for top 50%. These three together describe the variability rank distribution. |
| `expr_radar_*` | `eval_expr` | `med_prop_na` | Median proportion of samples where expression is missing (NA) across signature genes. Low is good. |
| | | `med_prop_above_med` | Median proportion of samples where a gene's expression exceeds the dataset-wide median expression threshold. High = genes are actively expressed. |
| `compact_radar_*` | `eval_compactness` | `autocor_median` | Median pairwise Spearman correlation between all pairs of signature genes. Values near 1 = genes move together (co-expressed). Values near 0 = genes are independent. A high-quality signature should have high autocorrelation. |
| `stan_radar_*` | `eval_stan` | `standardization_comp` | Spearman correlation between the sample scores computed from raw expression vs z-transformed expression. Values near 1 = z-transformation does not change the ranking of samples, which is desirable. Values near 0 = the signature is sensitive to expression scale, which is a concern. |
| `metrics_radar_*` | `compare_metrics` | `rho_mean_med` | Spearman correlation between the mean score and median score across samples. Near 1 = the two aggregation methods agree; outlier sensitivity is not a problem. |
| | | `rho_pca1_med` | Spearman correlation between PCA1 scores and median scores. Near 1 = the first principal component of signature gene expression explains the same sample ordering as the median score. |
| | | `rho_mean_pca1` | Spearman correlation between mean scores and PCA1 scores. |
| | | `prop_pca1_var` | Proportion of variance explained by the first principal component of the signature gene expression matrix. Near 1 = a single latent factor dominates the signature, which is the ideal for a coherent gene signature. |

### The full radar table: `radar_output_table.csv`

This file is the final output of the entire pipeline — one row per (signature, dataset) combination, 14 columns. The row index is `{signature_name}_{dataset_name}`.

The 14 columns are the union of all per-module radar metrics listed above. A cell value of `0` means the module did not run or the metric could not be computed (e.g., `prop_pca1_var` is 0 when `compare_metrics` is not included in the pipeline). All values are in `[0, 1]` after normalisation.

**How to read the table:** each row is a quality profile of one signature in one dataset. Rows with values close to 1 across all columns indicate a high-quality, well-behaved signature. Rows with low values on `autocor_median` indicate low co-expression. Rows with low `prop_pca1_var` indicate the signature lacks a single dominant signal. See `sigQC-master/AUDIT.md § ISSUE-2` and `§ ISSUE-3` for known limitations of the radar area summary.

## How cross-validation tests work

For each Python compute function, the cross-validation test:

1. Loads the fixture (same data the R reference was computed on).
2. Runs the Python function.
3. Loads the corresponding R reference CSV.
4. Calls `np.testing.assert_allclose(py_val, r_val, rtol=1e-4)`.

If Python and R disagree beyond `rtol=1e-4`, the test fails. Known sources of acceptable disagreement:

- **PCA sign ambiguity**: PCA eigenvectors are defined up to a sign flip. `rho_pca1_med` and `rho_mean_pca1` may have opposite sign between R and Python. Cross-validation tests use `abs()` for these two metrics. See `GUIDE.md` for details.
- **GaussianMixture vs Mclust**: EM initialisation differs between sklearn and R's mclust. Up to 5% relative tolerance is used for mixture model outputs.
